## Context

Текущий парсер — CLI-скрипт на Python + Telethon. Один пользователь, один чат, JSON-файлы. Превращается в SaaS-сервис с PostgreSQL.

**Стек**: Python, asyncpg/SQLAlchemy, PostgreSQL 15+, Telethon

**Схема обсуждалась и утверждена в explore-сессии.**

## Goals / Non-Goals

**Goals:**
- Полная реляционная схема для мульти-клиентного парсинга
- Учёт запросов (подписки + аддоны) атомарно на уровне БД
- Гибкая система каналов (один канал могут парсить N клиентов)
- Фильтрация по ключевым словам с записью срабатываний
- Class Table Inheritance: общая таблица `products` + специализированные `subscription_plans` / `addon_plans`

**Non-Goals:**
- UI / веб-интерфейс (только слой данных)
- Миграция старых данных (их нет)
- Горизонтальное масштабирование парсеров

## Decisions

### 1. Products + Class Table Inheritance
```
products ◄─ 1:1 ── subscription_plans
products ◄─ 1:1 ── addon_plans
```
**Почему:** единая точка для цены, названия, dodo_product_id. Специфичные поля — в дочерних таблицах.

### 2. Snapshot-поля total_requests в subscriptions / addon_purchases
**Почему:** историческая точность — изменение `request_limit` в плане не должно менять лимиты уже купленных подписок. Осознанная денормализация.

### 3. Атомарный инкремент used_requests
**UPDATE subscriptions SET used_requests = used_requests + 1 WHERE ...** — без race condition в asyncio. Никакой логики read→increment→write на уровне приложения.

### 4. Приоритет списания: подписка → аддоны
В одной транзакции: сначала списать с подписки, при остатке = 0 — списать с аддона. При исчерпании всего — блокировка клиента.

### 5. Один Telegram-аккаунт для всего
Все каналы парсятся через один Telethon-клиент (на этапе v1).

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Race condition списания с подписки и аддона в разных транзакциях | Единая хранимая процедура или SERIALIZABLE-транзакция |
| matches растёт бесконтрольно | Retention policy (удалять старше N дней), партиционирование по month |
| Username канала без нормализации → дубли | Тримминг @ и https://t.me/ перед записью |
| Один Telegram-аккаунт — лимиты API (flood wait) | Очередь запросов, планировщик |
